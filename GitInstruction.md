# Работа с Git

## 1. Проверка наличия установленного Git.
В терминале выполнить команду ` git version `.
Если Git установлен, появится сообщение с информацией о версии программы.
Иначе будет сообщение об ошибке.

## 2. Установка Git.
Загружаем последнюю версию Git с сайта https://git-scm.com/downloads. Устанавливаем с настройками по умолчанию.

## 3. Настройка Git.
При первом использовании Git необходимо представиться.
Для этого нужно выполнить в терминале две команды:
```
git config --global user.name «Ваше имя английскими буквами»
git config --global user.email ваша почта@example.com
```
## 4. Инициализация репозитория.
Чтобы Git начал отслеживать все, что будем делать в нашей папке, необходимо выполнить команду ` git init `. 
![git init](git_init.png)
В папке создается скрытая папка *.git*.
В ней ничего менять не надо. Это специальная папка для работы Git.

Дополнение: Чтобы на любом этапе узнать, что Git думает о нашей папке и её содержимом, а также статус происходящего, существует команда ` git status `.

## 5. Запись изменений в репозиторий.
Добавляем файл к будущему коммиту. Для этого необходимо выполнить команду  ` git add <имя_файла> `.

Теперь сохраняем изменения и фиксируем текущий статус. Для этого вызываем команду ` git commit -m "<комментарий>" `.
Обязательно указываем комментарий, чтобы в
дальнейшем понять, какое было сделано сохранение и для чего.

## 6. Просмотр истории коммитов.
Чтобы просмотреть какие версии существуют, используется специальная команда ` git log `. Это просто журнал изменений. На экран выводится истории всех коммитов с их хеш-кодами.
![git log](git_log.png)

` git log ` имеет огромное множество дополнительных параметров, которые будут влиять на вывод в консоль. Например, ` git log --oneline` - записывает каждый коммит в одну строку; ` git log --graph ` позволяет отобразить коммиты в виде дерева.

## 7. Перемещение между сохранениями.
* Чтобы перейти к какой-то версии, сохранению, надо
вызвать команду ` git checkout <первые четыре символа xeш-кода сохранения, которое надо загрузить>`.
* Команда ` git checkout master ` позволит вернуться к актуальному состоянию и продолжить работу. 
* Команда ` git diff ` покажет разницу между текущим файлом и закоммиченным файлом.

## 8. Игнорирование файлов.
Для того, чтобы исключить из отслеживания репозитории определенные файлы или папки необходимо создать файл ***.gitignore*** и записать в него их названия или шаблоны, соответствующие таким файлам или папкам.

## 9. Создание веток в Git.
Ветка в Git - это простой перемещаемый указатель на один из коммитов, обычно последний в цепочке коммитов.
По умолчанию имя основной ветки в Git - *master*.  

Создать ветку можно командой
```
 git branch <имя новой ветки>
```
В результате создается новый указатель на текущий коммит.

Эта команда только создает ветку, но не изменяет указатель на текущую ветку. Для того, чтобы сделать ветку текущей, нужно выполнить команду:
```
git checkout <имя ветки>
```
Чтобы создать ветку и сразу переключиться на нее, можно выполнить команду git checkout с параметром -b:
```
git checkout -b <имя новой ветки>
```

Список веток в репозитории можно посмотреть с помощью команды
```
git branch
```
Текущая ветка будет отмечена звездочкой: **\*master**

## 10. Слияние веток и разрешение конфликтов.
Для слияния выбранной ветки с текущей нужно выполнить команду:
```
git merge <название выбранной ветки>
```
Если была изменена одна и та же часть файла в обеих ветках, то может возникнуть конфликт, который потребует участия пользователя. VSCode предлагает варианты разрешения.
Чтобы разрешить конфликт, придётся выбрать один из вариантов, либо объединить содержимое по-своему.
Если вас все устраивает и вы убедились, что все файлы, где были конфликты, добавлены в индекс - выполните команду ` git commit ` для создания коммита слияния.

## 11. Удаление веток.

Если ветка больше не нужна, ее можно удалить, выполнив команду:
```
git branch -d <название удаляемой ветки>
```
-d - сокращение от delete (удаление).

![delete branch](deleteBranches.png)

Какие могут возникнуть ошибки при удалении ветки:
* нельзя удалить ветку, в которой вы находитесь и которую просматриваете в данный момент.
* если в ветке присутствуют несмерженные изменения или незапушенные коммиты, команда ` git branch -d ` не позволит удалить такую локальную ветку.

Типичная последовательность действий, которой завершаентся над какой-либо задачей проекта в отдельной ветке (назовем ее для примера *task*), будет такой:
```
git checkout master
git merge task
git branch -d task
```

## Полезная информация:
* Чтобы не писать команды заново, можно воспользоваться  стрелочками «вверх» и «вниз».
* Чтобы не писать название файла или ветки целиком, достаточно набрать первые два символа и нажать Tab, терминал самостоятельно заполнит то, что надо написать.

# Работа с удалёнными репозиториями

GitHub — это сервис компании Microsoft,который позволяет интегрироваться с программой Git и настроить удалённую работу с репозиториями.

GiHub:
* Самый популярный сервис Git
* Много полезных функций
* Огромный архив различного
кода

## Работа со своими репозиториями:
1. Создать аккаунт на GitHub.com.
2. На GitHub создать удаленный репозиторий.
3. Создать локальный репозиторий.
4. «Подружить» ваш локальный и удалённый репозитории. *GitHub при создании нового репозитория подскажет, как это можно сделать.* 
Для информации: для того чтобы убедиться, что репозитории связаны: 
* команда ` git remote` показывает, какие удаленные репозитории присоединены.
* команда ` git remote -v` выведет список с именами и соответствующими url-адресами репозиториев. 
5. Отправить (push) ваш локальный репозиторий в удалённый (на GitHub).
Отправить локальную версию репозитория во внешний репозиторий поможет команда ` git push `. При первом её использовании нужна
авторизация.
6. Провести изменения «с другого компьютера». Можно сделать это на GitHub.
7. Выкачать (pull) актуальное состояние из удалённого репозитория.
 Команда: ` git pull ` выкачивает данные из удаленного репозитория и делает слияние с локальным  репозиторием. Эту команду рекомендуется использовать внимательно, так как возможны ошибки как на этапе загрузки коммитов в локальную копию удаленной ветки, так и на этапе слияния веток.

## Работа с чужими репозиториями:

**pull request** - инструмент в Git, который позволит другим людям предлагать изменения в проект (request — «запрос»). Таким образом, **pull request** — это запрос на вливание в какой-то репозиторий. 
Когда хотим внести или предложить изменения в чей-то проект, это всегда делается в отдельной ветке.

**Порядок действий:**
1. В GitHub делаем копию интересующего нас репозитория. Для этого используется
кнопка **Fork**. 
2. Создаем копию этого репозитория на компьютере для нашей версии, используя команду: 
```
git clone <url-адрес репозитория>
```
Для информации: cкачать чужой репозиторий можно без создания собственного аккаунта.

3. Создаём ветку с предлагаемыми изменениями.
4. Производим все изменения только в этой ветке.
5. Отправляем эти изменения на свой аккаунт (push).
6. В окне на GitHub появляется возможность отправить **pull request**.
