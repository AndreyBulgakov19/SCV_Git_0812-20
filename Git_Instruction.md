# Работа с Git.
## 1. Проверка наличия установленного Git.

В терминале выполнить команду `git version`. Если Git установлен, то появиться сообщение с информацией о версии программы. 
Иначе появиться сообщение об ошибке.

## 2. Установка Git. Ссылка.

В пояснении используем ссылку на ресурс Git, где загружаем последнюю версию. Делается это вот так
[заголовок](сама ссылка): [Git](https://git-scm.com/download). Скачиваем и устанавливаем с настройками по умолчанию.

## 3. Настройка Git.

При первом использовании Git необходимо представиться. Для этого выполним две последующие команды:
```
git config --global user.name "ваше имя латиницей"
git config --global user.email ваша почта@example.com
git config --global --list
```
Последняя команда используется используется для проверки.

 
 ## 4. Инициализация репозитория.

Чтобы начать отслеживание за проетом необходимо выполнить команду `git init`.
 Эта команда создает в текущем каталоге новый подкаталог с именем `.git`, содержащий все необходимые файлы репозитория - структуру Git репозитория. 
 На этом этапе проект ещё не находится под версионным контролем. Данную команду необходимо выполнять самой первой.

  ## 5. Запись и изменение в репозитории.
 ### Проверим статус файла и используем цитату. 
 
 Когда есть необходимость выделить некий абзац как цитату, то используется знак `>`. Его необходимо проставлять перед каждым элементом цитаты. Обычно используется, чтобы показать исходный текст. Далее используем этот инструмент.
 Основным инструментом для определения статуса файлов является команда `git status`. Сразу после клонирования получается примерно такой результат:

  >git status

 >On branch master

 >No commits yet
 
 >Untracked files:

 >(use "git add <file>..." to include in what will be committed)

 >       Git_Instruction.md

 >nothing added to commit but untracked files >present (use "git add" to track)

 Это означает, что файлы в рабочем каталоге не отслеживаются и не изменяются.
Кроме того, Git нашел неотслеживаемый файл ***Git_Instruction.md***.
Наконец, команда сообщает имя ветки, в которой мы сейчас находимся.
Пока мы будем работать только с веткой ***master*** по умолчанию.
Ещё предлагается ввести команду `git add` для отслеживания.
Для вывода сокращенной информации к команде `git status` добавляется флаг `-s`.

### Отслеживание новых файлов.
Чтобы начать отслеживать новый файл, используем `git add`. Например, файл ***Git_Instruction.md*** будет выглядеть так:
`git add Git_Instruction.md`
Команда состояния теперь покажет, что этот файл отслеживается и индексируется.
Проверяем статус через команду: `git status` и теперь появилась информация о том, что есть изменения для фиксации. 
Нужный нам файл теперь подсвечен зеленым цветом. Но файл еще находится в промежуточном состоянии.
Для добавления всех файлов в репозитории используется команда `git add .`.

### Зафиксировать изменения.

 Теперь, когда область индексирования настроена в соответствии с нашими потребностями, мы можем зафиксировать изменения.
Самый простой способ фиксации — с помощью `git commit`.
Эта команда открывает выбранный нами текстовый редактор.
Мы также можем установить фиксацию (в команде фиксации) добавив флаг `-m` перед сообщением. Например `git commit -m "initial commit"`. 
При добавлении флага -a выполним и подготовку файла и фиксацию.
Есть другой способ фиксации. После ввода команды `git commit -a` мы попадаем в текстовый редактор. 
Теперь мы находимся в командном режиме. 
Для перехода в режим редактирования надо нажать на клавишу `i`. Вводим текст. 
После нажатия клавиши `q` мы вновь попадаем в командный режим. Вводим `wq:` и далее `enter`. Можно установить свой редактор через файл конфигурации.

### Просмотр индексированных и неиндексированных изменений.

Если команда git status не предоставляет достаточно подробностей, например, если мы хотим получить не только список отредактированных файлов,
но также чтобы узнать, что изменилось, мы будем использовать `git diff`.
Команда `git status` отвечает на эти вопросы в общих чертах, перечисляя имена файлов,
но команда `git diff` показывает добавленные и удаленные строки, то есть все вставки в программу.

 ### Просмотр истории сохранений.
 
После сохранения нескольких версий файлов или клонирования существующего репозитория
есть возможность взглянуть на то, что было сделано ранее.
Базовым и наиболее мощным инструментом в данном случае является команда `git log`.
Одним из наиболее полезных является параметр -p, который показывает разницу, сделанную каждой версией сохранения.
Опция -2 ограничивает вывод двумя последними записями.
Для минимального вывода информации используется команда `git log --oneline`.

 ### Перемещение между сохранениями.

Команда `checkout` может использоваться для переключения между сохранениями.
Мы можем перейти к началу ветки `git checkout` или к отдельному коммиту `git checkout sha`.
`git checkout master` возвращает текущее состояние. Команда аналогичного действия `git checkout swith` для возврата в исходное состояние. 

## 6. Добавление изображения.

Для добавления изображения необходимо файл ,содержащий нужное нам изображение, добавить в корневую папку. В файле с разметкой Markdown и расширением .md, в нужном нам месте, расположим текст с указанием названия и относительного пути к изображению.
Выглядеть это будет вот так: <p><font size="3" color="#fa8e47" face="serif"> ![название файла](относительный путь).</font></p>


## 7. Игнорирование файлов.

Для того, чтобы исключить из отлеживания в репозитории определённые файлы или папки необходимо создать файл ***.gitignore*** 
и записать в него названия файлов или лучше шаблоны соответствующие этим файлам или папкам.
Например: 
```
*png
   
*jpg
   
*bmp

*gif
```

Можно так же добавить к исключению и папки:

`/<имя папки>`

Выполняем команду для добавления к отслеживанию:
`git add .gitignore`.


## 8. Создание веток в Git.

Ветка в Git - это простой перемещаемый указатель на один из коммитов, обычно последний в цепочке коммитов. По умолчанию имя основной ветки в Git - *master*.

Создать ветку можно одной из двух команд:
```
git branch <имя новой ветки>

или

git swith -c <имя новой ветки>
 ```
В результате создаётся новая ветка.

Список веток в репозитории можно помотреть с помощью команды:
```
git branch
```
Текущая ветка будет отмечена звёздочкой: 

*\*<имя ветки>*.
 
Смотрим изменения вводя команду с визуализацией:

`git log --graph` 

или короткое содержание 

`git log --oneline`.

Создаем новую ветку:

 `git branch first`.

Переходим на новую ветку:

`git checkout first`.
 
Редактируем текст. Сохраняем коммит. 
Переходим в основную ветку:

`git checkout master`. 
 
 Флаг `-b` позволяет сразу перейти в созданную ветку:

`git branch -b <имя ветки>` 

Вводим команду для просмотра изменений:

`git log --graph` 

или для сжатого изображения:

`git log --oneline`.


## 9. Слияние веток и разрешение конфликтов.

Создаём три новых ветки:

`git branch second`

`git branch third` 

`git branch fourth`

Для слияния выбранной ветки с текущей нужно выполнить команду:
```
git merge <название выбранной ветки>
```
Вызывать командой надо из той ветки в которую будет происходить вливание из другой .
 
Если была изменена одна и таже часть файла в обеих ветках, то может возникнуть конфликт, который потребует участие пользователя. 
VSCode предлагает варианты разрешения:

а) принять существующий вариант;
б) принять входящий вариант;
в) принять оба вариента.

После разрешения конфликта выполнится слияние.


## 10. Удаление веток.

Удаление ветки произойдёт после ввода команды:

 `git branch -d <название ветки>`
 
Далее удаляем ненужные ветки `first`, `second`, `third`, `fourth`:

`git branch -d first`

`git branch -d second`

`git branch -d third`

`git branch -d fourth`

 Возможно сначала VSCode выдаст предупреждение о потере данных и предложит команду на удаление:

$ git branch -D <имя ветки>

Для удаления ветки необходимо перейти на ветку `*master`. Только после этого можно удалить черновик, иначе будет ошибка:

$ git branch -D <имя ветки>
error: Cannot delete branch 'имя ветки' .
